Making the hierarchy diagram mathematically smarter

Seven proposals for normedfield_hierarchy.html that surface Mathlib design knowledge rather than just structure syntax. Code snippets below are verified against the local mathlib4 checkout (v4.31.0 toolchain).

accuracy1 · Mark default-valued fields — npow is not extra data

The tooltip for Monoid currently lists npow, npow_zero, npow_succ as if they were data an instance author must supply, on par with one_mul. They are not. In Mathlib’s actual definition every one of them has a default value:

class Monoid (M : Type u) extends Semigroup M, MulOneClass M where
  protected npow : ℕ → M → M := npowRecAuto          -- n-fold product, the “natural definition”
  protected npow_zero : ∀ x, npow 0 x = 1 := by intros; rfl
  protected npow_succ : ∀ (n : ℕ) (x), npow (n + 1) x = npow n x * x := by intros; rfl

Why the field exists at all: if x ^ n were defined as iterated multiplication, then for ℝ the monoid power and the native fast power would be different terms and definitional-equality diamonds would appear downstream. Storing npow as an overridable field with a default is Mathlib’s standard fix (same pattern: nsmul, zpow, div := fun a b => a * b⁻¹, DivisionRing.nnratCast := NNRat.castRec, …). Separately, Monoid.toPow is declared @[default_instance high] — which is probably what “default instance” was gesturing at; both facts are worth showing.

content2 · Distinguish data from laws, mixins from structures

Two orthogonal distinctions that the diagram currently flattens:

Field level. Each field is either data (norm : α → ℝ, npow) or a law (a Prop, like norm_mul). Badge them, as in the mockup above — it makes “what structure am I carrying vs. what am I proving” visible at a glance. Checking is trivial in the generator: is the field’s type a Prop?

Class level. Prop-valued classes (T2Space, Nontrivial, IsDomain, CompleteSpace…) are mixins: they add no data, only a theorem about existing structure. Data-carrying classes (Monoid, NormedField…) are honest structures. Figure 1 of the CPP ’20 paper drew these with different node shapes; the diagram should too — e.g. rounded rectangles for data classes, ellipses (or a dashed border) for Prop mixins. The information is already in CLASS_DATA[*].type (it ends in … → Prop).

Mockup — NormedField tooltip with both badges

NormedField data class
Adds over parents (Norm, Field, MetricSpace)
  • dist_eq : dist x y = ‖-x + y‖ law — glues the metric to the norm
  • norm_mul : ‖a * b‖ = ‖a‖ * ‖b‖ law — the norm is multiplicative, not just submultiplicative

Read this way, NormedField is revealed as: a field, a metric space, and a norm, plus two compatibility laws. That is the actual mathematical content, and it is invisible in the current rendering.

Data needed: one isProp boolean per field and per class; both are one-line checks in the generator.

content3 · Example types and separating witnesses

For a mathematician the fastest way to understand a class is an example — and the fastest way to understand an edge is a counterexample showing the inclusion is strict. Both can be computed mechanically: fix a roster of ~40 standard types and, in the generator, attempt #synth C T for every (class, type) pair. Embed the resulting boolean matrix (~4k bits) in the page.

Per-node: “examples” line, sharpest first

Monoid

List α, Matrix (Fin 2) (Fin 2) ℝ — go no higher in this diagram (not Comm, not Group, no zero¹)
also: ℕ, ℤ, ℝ, ℂ, Equiv.Perm (Fin 3), … (which are also stronger classes)

¹ “Go no higher” = the type inhabits this class but none of the classes with an in-diagram edge into it. List α under ++ is the free monoid: associative with unit, non-commutative, no inverses, no zero — it uniquely conforms to Monoid among the drawn classes. That single example teaches more than the field list does.

Per-edge: strictness witnesses

Hovering the edge Field → CommRing (say) shows a type that is a CommRing but not a Field:

Field → CommRingℤ (no inverses), ZMod 6 (zero divisors)
Field → DivisionRingℍ quaternions (non-commutative)
CommMonoid → MonoidList α, Equiv.Perm (Fin 3)

The last row shows the honest caveat this feature must carry: instance search measures “does Mathlib declare it”, not “can it exist”. For Prop mixins that caveat disappears and witnesses are genuine counterexamples — Mathlib even ships a Counterexamples/ library worth linking for hand-curated ones (e.g. T1-not-T2 needs a bespoke space like the cofinite topology, which no roster type will exhibit).

Data needed: a Lean script that loops Class × RosterType and records instance-synthesis success (seconds of CPU; entirely mechanical). Bonus: it doubles as an empirical check of the diagram — every drawn edge S→T should satisfy “instances(S) ⊆ instances(T)” over the roster.

content4 · Give every edge its name

Every arrow in the diagram is backed by a real declaration, but the page keeps them anonymous. Solid edges are projections (Monoid.toSemigroup — already stored in parents[].projection, shown only in the node panel); dashed edges are instances with names, priorities, and docstrings:

instance (priority := 100) T2Space.t1Space [T2Space X] : T1Space X := …
@[default_instance high]
instance Monoid.toPow {M : Type*} [Monoid M] : Pow M ℕ := ⟨fun x n ↦ Monoid.npow n x⟩

Make edges hoverable/clickable like nodes: show the declaration name, its priority (which explains instance-resolution order — a genuinely useful thing to see), the docstring, and a link to source / Mathlib docs. The existing incident-edge hover highlighting already proves the hit-targets work.

Data needed: for dashed edges, the instance name found during generation (the generator already verified these against source, so it has them); add an EDGE_DATA map keyed by "S->T". UI: reuse the existing tooltip machinery on g.edge elements, with fatter invisible hover strokes for clickability.

content5 · Explain the diamonds — forgetful-inheritance callouts

The subtlest design idea in this hierarchy is invisible in the picture: some solid edges carry data with a default precisely to keep competing paths definitionally equal. The canonical case is right in this diagram:

class PseudoMetricSpace (α : Type u) extends Dist α where
  dist_self, dist_comm, dist_triangle                    -- the 3 fields you must give
  edist : α → α → ℝ≥0∞ := fun x y => …
  toUniformSpace : UniformSpace α := .ofDist dist …     -- ◇ data + default
  uniformity_dist : 𝓤 α = ⨅ ε > 0, 𝓟 {…} := by intros; rfl
  toBornology : Bornology α := Bornology.ofDist dist …  -- ◇ data + default
  cobounded_sets : … := by intros; rfl

Why: ℝ × ℝ gets a uniformity from the product of uniform spaces and from its metric. If toUniformSpace were derived rather than stored, those two would be different terms and nothing typechecks across them. Storing it as an overridable field (with a propositional guard, uniformity_dist) makes the diamond commute definitionally. This is the “forgetful inheritance” pattern (Affeldt–Cohen–Kerjean–Mahboubi–Rouhling–Sakaguchi), discussed at length in the CPP ’20 paper the diagram is modelled on.

Proposal: mark the affected edges (PseudoMetricSpace → UniformSpace, → Bornology, PseudoEMetricSpace → UniformSpace, SeminormedRing → PseudoMetricSpace’s metric-vs-norm glue, Monoid → NPow, …) with a ◇ badge, and give them a shared explainer panel — one paragraph, shown for whichever ◇ edge is clicked, with the class-specific default spelled out. This turns the diagram into a teaching tool for the one idea every new Mathlib contributor trips over.

Data needed: nothing new — the ◇ set is exactly “solid edge whose projection target is a data field with a default value”, which falls out of proposal 1’s extraction.

content6 · “How do I build one?” — constructors in the panel

The pinned panel says what a class is but not how instances are made in practice. Mathlib has strong conventions worth surfacing; for NormedField the real-world answers are:

NormedField
Ways to build an instance
  • ⟨dist_eq, norm_mul⟩ directly, given Field + Norm + MetricSpace
  • NormedField.induced — pull back along an injective ring hom f : R →+* S into a NormedField
  • SubfieldClass instance — subfields inherit automatically (uses .induced)

Pullback constructors (*.induced, Function.Injective.*) are the workhorses of instance building and follow a naming convention, so they can be found mechanically: scan the environment for declarations whose result type is C _ with hypotheses mentioning an injection or a hom. Even a curated list for the ~15 “headline” classes (NormedField, MetricSpace, UniformSpace, Field, Group…) would carry most of the value.

Data needed: a generator pass over declarations returning each class (filter by naming convention .induced/Injective./.ofDist-style), or a hand-curated YAML for headline classes. Rendered as one extra section in the pinned panel.

interaction7 · “Why is every X a Y?” — two-node path explanation

The relation highlighting added recently shows that everything green is derivable; the natural next question is how. Proposal: pin a node, then shift-click a second node in its cone. The page highlights the actual derivation path(s) and the panel lists the chain by name (using proposal 4’s edge names):

NormedField ⇒ T2Space, because:
  • NormedField.toMetricSpace (projection)
  • MetricSpace.toEMetricSpace → EMetricSpace.instT2Space (derived instances)
2 of 3 drawn paths shown · shortest first

This answers the question people actually bring to this diagram (“why does my normed field magically have a Hausdorff topology?”) and it exercises the mathematics of the picture rather than its geometry. Implementation is pure JS on the already-parsed edge adjacency: BFS all shortest paths, reuse the fade/highlight CSS, no new data (nicer with proposal 4’s names).
